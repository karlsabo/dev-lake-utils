# Add JetBrains changelog Gradle plugin

## Goal

Make changelog maintenance available through Gradle by adding the JetBrains Gradle Changelog plugin (`org.jetbrains.changelog`) to the root build.

## Context

- Target plugin: JetBrains Gradle Changelog Plugin, https://github.com/JetBrains/gradle-changelog-plugin, plugin id `org.jetbrains.changelog`.
- The root Gradle build is `dev-lake-utils.gradle.kts`; `settings.gradle.kts` sets `rootProject.buildFileName = "dev-lake-utils.gradle.kts"`.
- Plugin versions and aliases are centralized in `gradle/libs.versions.toml` under `[versions]` and `[plugins]`.
- The root build already applies version-catalog plugin aliases for `spotless` and `detekt` in `dev-lake-utils.gradle.kts`.
- `settings.gradle.kts` includes `gradlePluginPortal()` in `pluginManagement.repositories`, so Gradle can resolve the JetBrains changelog plugin.
- The root project currently sets `version = "0.1.0-SNAPSHOT"` in `dev-lake-utils.gradle.kts`.
- No `CHANGELOG.md` exists at the repository root yet.

## Decisions

- Add the JetBrains Gradle Changelog Plugin from https://github.com/JetBrains/gradle-changelog-plugin.
- Pin the latest stable plugin version available at implementation time.
- Run the plugin's changelog initialization task and commit the generated root `CHANGELOG.md`.
- Use JetBrains default changelog groups/formatting.
- Do not add special snapshot-version handling. While the project version is `0.1.0-SNAPSHOT`, the plugin should keep using its in-progress/current changelog section semantics.

## Acceptance tests

1. Given the repository after the PR, when a maintainer runs `./gradlew getChangelog`, then Gradle reads the initialized root `CHANGELOG.md` through the JetBrains changelog plugin without configuration errors.

## Stories

### 1. Add and initialize the JetBrains changelog Gradle plugin

**Acceptance criteria:** Given the repository after the PR, when a maintainer runs `./gradlew getChangelog`, then Gradle reads the initialized root `CHANGELOG.md` through the JetBrains changelog plugin without configuration errors.

**Expected edits:**

- `gradle/libs.versions.toml`: add the latest stable JetBrains changelog plugin version and plugin alias.
- `dev-lake-utils.gradle.kts`: apply `alias(libs.plugins.jetbrains.changelog)` and keep configuration minimal, using JetBrains defaults unless required for the existing project version.
- `CHANGELOG.md`: add the initialized changelog file generated by the plugin's initialization task.

**Scope:**

- In: plugin declaration, root application, initialized changelog file, JetBrains default changelog format/groups.
- Out: release automation, GitHub release publishing, CI changes, generated release notes, historical changelog backfill, custom snapshot-version handling.

**Notes:**

- Follow the existing plugin-management pattern in `gradle/libs.versions.toml` rather than hard-coding the plugin version in `dev-lake-utils.gradle.kts`.
- After applying the plugin, run its init task, likely `./gradlew initializeChangelog`, and commit the generated `CHANGELOG.md`.
- Validate with `./gradlew getChangelog`, `./gradlew tasks --all`, and `./gradlew check`.
